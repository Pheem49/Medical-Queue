คำอธิบายการทำงานของฟังก์ชัน (Service Functions) ทั้งหมดในไฟล์ `services/booking_service.py`
ไฟล์นี้ทำหน้าที่เป็นชั้น "Business Logic" สำหรับจัดการ คิวจอง, ยกเลิก, เลื่อนคิว และดึงประวัติการจองของผู้ป่วย 

======================================================================
1. encrypt_national_id(national_id)
- ทำหน้าที่: ฟังก์ชันเข้ารหัสเลขบัตรประชาชน 13 หลัก ให้กลายเป็น Token (สร้างเป็น QR Code) เป็นระบบความปลอดภัยไม่ให้ข้อมูลรั่วไหล
- Database/ข้อมูล: ไม่มีการแก้ข้อมูลฐานข้อมูล เพียงรับ String (`national_id`) มาเข้าสมการ `Fernet` (AES encryption) แล้วคืนค่า String กลับไปใช้งาน
- อยู่ส่วนของหน้าไหน: ทำงานเบื้องหลังการเชื่อมต่อ API (Background API) ตอนกดจองคิว
- ทำส่วนไหน: เข้ารหัสข้อมูลให้ปลอดภัยก่อนแปลงและตอบกลับออกไปในหน้าการ์ดบัตรคิว (QR Code)
- ไอคอน / ปุ่ม: ไม่มี ทำงานอัตโนมัติ

2. decrypt_national_id(token)
- ทำหน้าที่: ไว้ใช้ถอดรหัสฝั่งเจ้าหน้าที่ (เมื่อสแกน QR Code) เพื่อดึงเลขบัตรประชาชนจริงๆ คืนมา
- Database/ข้อมูล: รับ String ที่เข้ารหัส และพ่น String บัตรประชาชนที่ถอดรหัสแล้วออกมา (หรือ None หากเกิด Error)
- อยู่ส่วนของหน้าไหน: ฝั่งระบบของเจ้าหน้าที่
- ทำส่วนไหน: ถอดรหัสหลังจากการสแกนเพื่อยืนยันตัวตน
- ไอคอน / ปุ่ม: เป็นการทำงานผ่านหน้าจอ/เครื่องสแกนของเจ้าหน้าที่แบบเบื้องหลัง

3. get_active_booking(user_id)
- ทำหน้าที่: ค้นหาว่าผู้ป่วยคนนี้มีคิวจองที่กำลังรอรับบริการค้างอยู่ไหม (เพื่อไม่ให้กดจองซ้ำซ้อน) *นอกจากนี้ยังแฝงการป้องกัน "คิวหมดอายุ" โดยถ้าเลยเวลาสิ้นสุดของคิว ระบบจะยกเลิกคิวนี้อัตโนมัติ*
- เชื่อมไปไฟล์ไหน: ถูกเรียกใช้ประจำจาก `views/booking_routes.py` (หน้า /booking, /booking/confirm เป็นต้น)
- Database:
  > (Read): ค้นหาโมเดล `Booking` ที่สถานะเป็น 'รอรับบริการ'
  > (Update/Auto-Cancel): หากเวลาปัจจุบันเลย `slot.end_time` ไปแล้ว จะอัปเดตสถานะ `Booking` = 'ยกเลิก' และหักของช่องตารางคิว (`AppointmentSlot`) คืนให้ระบบพร้อมอัปเดตข้อมูล
- อยู่ส่วนของหน้าไหน: ในจังหวะกำลังโหลดเข้าหน้า `booking.html` หรือ `confirm_booking.html`
- ทำส่วนไหน: ทำงานตรวจนับตอนโหลดหน้าเว็บ เพื่อบล็อกไม่ให้คนไข้กดจองซ้อนทับกัน
- ไอคอน / ปุ่ม: ทำงานเองอัตโนมัติรอดักอยู่ก่อนโชว์หน้า UI

4. create_booking(user_id, slot_id, detail)
- ทำหน้าที่: พระเอกในการ "สร้างการจองคิวใหม่" อย่างเป็นทางการ
- Database:
  > (Read): เช็คจาก `get_active_booking`, หา `AppointmentSlot` มาเช็คว่าเต็มไหม (`max_capacity`), ดึงข้อมูล `User` เพื่อเอาเลขบัตรประชาชน
  > (Read/Calculate): หาเลขคิวประจำวัน (Queue Number) ด้วยคำสั่ง `func.max` ของแผนก+วันที่นั้นๆ
  > (Create/Update): หักที่นั่งจอง (เพิ่ม `current_booking`), เซฟการจองใหม่ลง `Booking` พร้อมรหัส `qr_code` 
- ข้อมูลส่ง/รับ: รับ `user_id`, `slot_id`, `detail` สั่งเช็คตรวจสอบ ถ้าสำเร็จ Commit (เซฟ) ฐานข้อมูลส่งกลับเป็น Dictionary แจ้ง "success" หรือ "error"
- อยู่ส่วนของหน้าไหน: รับคำสั่งมาจากหน้า `confirm_booking.html`
- ทำส่วนไหน: นำชุดข้อมูลส่งเข้าเซิร์ฟเวอร์เพื่อ Commit ลงกระดานฐานข้อมูล 
- ไอคอน / ปุ่ม: ตอบสนองเมื่อคนไข้กด **ปุ่ม "ยืนยันการจองคิว"** (ปุ่มสี่เหลี่ยมทึบสีน้ำเงินใหญ่ รูปแบบ Confirm บันทึก)

5. get_patient_history(user_id)
- ทำหน้าที่: ดึงรายการประวัติการจองทั้งหมดของผู้ป่วยคนเดียว เรียงจากใหม่ไปเก่า
- Database:
  > (Read/Join): คิวรี่รวบ 4 ตาราง คือ `Booking`, `AppointmentSlot`, `Doctor`, `Department` ให้ได้ทั้งชื่อหมอ แผนก เวลา คิว สถานะ แบบครบถ้วน
- ข้อมูลส่ง/รับ: แปลง Object คิวรี่ให้ออกมาเป็น List of Dictionary (Array) เพื่อส่งให้ API นำไปแสดงผล
- อยู่ส่วนของหน้าไหน: หน้า "บัตรคิวของฉัน" (`mytickets.html`) และ หน้า "ประวัติ" (`history.html`)
- ทำส่วนไหน: ประมวลรวบรวม Array ไว้ให้หน้าเว็บไปยิงลูปทำกล่องการ์ด
- ไอคอน / ปุ่ม: ควบคุมหน้าเว็บโหลดข้อมูลเองอัตโนมัติ

6. cancel_booking(user_id, booking_id)
- ทำหน้าที่: ผู้ป่วยขอยกเลิกนัดจอง
- Database: 
  > (Read): หาตั๋วการจอง `Booking`
  > (Update): เติมลอจิกเปลี่ยนสถานะเปลี่ยนเป็น 'ยกเลิก' และคืนปริมาณคิว (`current_booking -= 1`) ตัว `AppointmentSlot` สามารถกลับมาเป็นป้ายสถานะ 'active' ได้ถ้าเดิมขึ้น 'เต็ม' อยู่
- ข้อมูลส่ง/รับ: รับไอดีผู้ใช้และไอดีการจอง ถ้าจัดการสำเร็จก็ส่งกลับ Dictionary success
- อยู่ส่วนของหน้าไหน: หน้าประวัติคิวปัจุบัน `mytickets.html` 
- ทำส่วนไหน: สั่งปลดล็อคยอดจองออกคืนคิวกลับสู่ระบบปกติ
- ไอคอน / ปุ่ม: เมื่อกดที่ **ปุ่ม "ยกเลิกคิว"** (ปุ่มกรอบเส้นแดง ตัวอักษรสีแดง เล็กๆ อยู่ใต้การ์ดบัตรคิว)

7. get_booking_details(user_id, booking_id)
- ทำหน้าที่: ดึงรายละเอียดการจอง "แบบเจาะจง 1 รายการ" ไว้ไปแสดงผลในหน้าตั๋วของฉันหรือคิวอาร์โค้ดยืนยันเจาะลึก
- Database:
  > (Read/Join): ผูก 4 ตารางคล้ายตัวดึงประวัติ แต่ฟิลเตอร์หาเฉพาะ `Booking.id = booking_id`
- ข้อมูลส่ง/รับ: แปลง Object ที่ซับซ้อนไปเป็น Dictionary แตกสโคบออกมาเป็น Data JSON เรียบร้อย
- อยู่ส่วนของหน้าไหน: เป็น API ส่งผ่านข้อมูลเบื้องหลัง
- ทำส่วนไหน: กรองเอามาแค่หนึ่งใบรับไว้โหลดข้อมูลย่อย
- ไอคอน / ปุ่ม: ซ่อนอยู่เบื้องหลังไม่มีปุ่มตรงๆ

8. get_available_dates(doctor_id)
- ทำหน้าที่: ดึงลิสต์รายชื่อวันที่คุณหมอคนดังกล่าว "มีตารางให้จอง/ยังว่างอยู่" 
- Database:
  > (Read/Filter): เช็คโมเดล `AppointmentSlot` เฉพาะรายชื่อของหมอคนนั้น วันที่ในอนาคต (> วันนี้) และจำนวนการจองยังไม่เกินขีดจำกัด
- ข้อมูลส่ง/รับ: ส่งกลับเป็นลิสต์ของวันที่ในรูปแบบ YYYY-MM-DD แบบง่ายๆ ให้เอาไปทำปฏิทินเลือกวัน
- อยู่ส่วนของหน้าไหน: หน้ากรอกข้อมูลแรก `booking.html`
- ทำส่วนไหน: เติมลิสต์และอนุญาตให้วันที่นั้นๆ เปิดให้คนไข้กดได้ 
- ไอคอน / ปุ่ม: แสดงผลใน **ช่องปฏิทิน (วันที่)** โดยวันที่จองได้จะเป็นพื้นหลัง **สีฟ้า** บนฟิลด์ระบุวันนัดหมาย

9. reschedule_booking(user_id, booking_id, new_slot_id, new_detail)
- ทำหน้าที่: ระบบ "เลื่อนนัดหมาย"
- Database:
  > (Read): เช็คคิวเก่า, เช็คคิวใหม่ว่าเต็มไหม
  > (Update): "คืนที่นั่งเก่า (-1 ใน Slot เดิม)" และ "หักที่นั่งใหม่ (+1 ใน Slot ใหม่)"
  > (Update): คำนวณดึงเลขคิวลำดับใหม่ให้ของวันถัดไป และเปลี่ยนค่าตัวแปร `slot_id`,`detail`,`queue_number` บนแถวของ `Booking` แถวเดิม
- ข้อมูลส่ง/รับ: ดำเนินการ Commit อย่างซับซ้อน หากสำเร็จส่งผล Dictionary คอนเฟิร์มเลื่อนคิวสำเร็จ
- อยู่ส่วนของหน้าไหน: ในหน้า `confirm_booking.html` (โหมดสรุปการเลื่อนนัด)
- ทำส่วนไหน: วางระบบแทนที่ค่าของคิวเดิม ด้วยวันที่หริอเวลาคิวตัวใหม่ลง Database
- ไอคอน / ปุ่ม: จะบันทึกจริงๆ เมื่อรับกระแสคำสั่งมาจาก **ปุ่ม "ยืนยันการเลื่อนนัด"** (ปุ่มสีน้ำเงินใหญ่) ของหน้าสรุป 
- ปุ่มต้นทาง: เริ่มมาจากกการกด **ปุ่ม "เลื่อนนัด"** (ปุ่มคำว่าสีเหลือง กรอบสีเหลือง) จากหน้า `mytickets.html`

10. การเชื่อมโยงกับไฟล์หน้าเว็บ (Templates):
ไฟล์ Service นี้ไม่ได้รับผิดชอบเรื่องการเชื่อม HTML โดยตรง (ไม่มีการ render_template) แต่จะเป็นผู้ที่ "รับจบ" จัดการฐานข้อมูลหลังบ้าน ตัวอย่างคือ
- เมื่อหน้า `booking.html`, `confirm_booking.html`, `mytickets.html`, `history.html` ในโฟลเดอร์ \templates ส่งคำขอ (Request) ผ่านเส้น API 
- เส้นทาง Route ของ API จะเรียกใช้ฟังก์ชันในไฟล์ Service นี้ต่อ
- จากนั้นข้อมูลที่ Service คำนวณเสร็จ (เช่น คิวรี่ฐานข้อมูล/การสร้างข้อมูล) จะถูกตอบกลับไปในรูปแบบ Dictionary หรือตัวแปร และถูกแปลงเป็น JSON ส่งคืนให้กับหน้าเว็บ 4 ไฟล์เหล่านั้นนำไปแสดงผลครับ

สรุปการทำงานร่วมกัน:
- ไฟล์นี้ครอบคลุมฟังก์ชันการทำธุรกรรมแบบ CRUD เกือบทั้งหมด (ยกเว้นการลบจริง Delete เพราะเลือกใช้การอัปเดตสถานะเป็น "ยกเลิก" Soft Delete แทนเก็บ History ไว้)
- ตัวแปรฐานข้อมูล (Database) สำคัญสุดคือ `AppointmentSlot` กับ `Booking` ที่มักถูกใช้งานควบคู่เสมอ เพื่อเพิ่มลดปริมาณที่นั่ง (capacity logic) ให้ถูกต้องตลอดเวลา
- ทุกตัวที่เปลี่ยนแปลงค่า DB ปิดท้ายด้วย `db.session.commit()` เพื่อยืนยัน หรือ `db.session.rollback()` หาก Database เกิด Error ป้องกันข้อมูลพัง
